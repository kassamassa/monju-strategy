## 1. ビジョン

ユーザーの「野望」を入力として受け取り、
本質的な問いの発見から仮説検証・意思決定まで
戦略コンサルのプロセスを自動化するシステムを構築する。

壮輝自身がファーストユーザーとして使い、
Web制作×自動化事業の意思決定に適用する。
その過程でシステムの精度を上げ、将来SaaSとして外部販売する。

---

## 2. システムの本質

### 戦略コンサルのデファクトプロセスを自動化する

```
Step 1：問いの発見（Issue Identification）
  「本当に解くべき問いは何か」を特定する
  ※ ユーザーが言っている問題と本当の問題は違うことが多い

Step 2：仮説構築（Hypothesis Driven）
  答えを先に仮説として立てる
  「おそらく〇〇が原因で〇〇すれば解決する」

Step 3：検証設計（Analysis Design）
  仮説を証明・反証するために
  何のデータが必要かを先に決める
  ※ データを集めてから考えるのではない

Step 4：データ収集・分析
  外部API・SNS・市場データで仮説を検証する

Step 5：示唆の抽出（So What?）
  「だから何をすべきか」を導く

Step 6：意思決定支援
  優先アクションリストとロードマップを提示する
```

### 目標がぼんやりしていても機能する

このシステムは「目標を与えると戦略が出る」のではなく
「目標自体を発見する手助けをする」ことを核心とする。

野望だけある状態・目標がぼんやりしている状態から
本質的な問いを発見し、検証し、意思決定まで導く。

## 3. システム構成（4レイヤー）

### Level 0：問いの発見エンジン

```
入力：
  野望・現状・制約（自然言語で入力）
  例：「MONJUで成功したい。Web制作をやっているが
       将来は何か大きいことをしたい」

処理：
  ・ユーザーが言っている問題と本質的な問題を分離
  ・前提となっている仮定を明示化
  ・本当に解くべき問いを複数生成

出力：
  ・Issue Tree（問いの構造）
  ・本質的な問い候補リスト
  ・優先度スコア
```

### Level 1：仮説構築エンジン

```
入力：
  Level 0で選んだ本質的な問い

処理：
  ・MECEに仮説を展開する
  ・各仮説の前提条件を明示する
  ・仮説の優先順位を設定する

出力：
  ・検証すべき仮説リスト
  ・各仮説の重要度・検証難易度
  ・検証の優先順位
```

### Level 2：検証設計・データ収集エンジン

```
入力：
  Level 1で選んだ仮説

処理（外部API・データソース）：
  市場分析：
    PEST → 政府統計API・法令データ・ニュースAPI
    3C   → 国税庁法人番号API・競合分析（Claude API）
    STP  → 業種×規模×地域のスコアリング

  SNS分析：
    プラットフォーム特性（TikTok・Instagram・YouTube）
    バズっているコンテンツの形式・構造・タイミング
    視聴者属性・エンゲージメント率
    競合アカウントの分析

  競合分析：
    Web制作会社の集客方法・価格帯・訴求軸
    SerpAPI経由でGoogle検索結果を分析

出力：
  ・仮説の検証結果（支持/反証/不明）
  ・データソースと信頼度
  ・追加で必要なデータの提案
```

### Level 3：示唆抽出・意思決定支援エンジン

```
入力：
  Level 2の検証結果

処理：
  ・So What?（だから何をすべきか）を導く
  ・優先アクションをスコアリング
  ・ロードマップを生成
  ・KPIを設計する

出力：
  ・戦略レポート
  ・優先アクションリスト（即実行可能な順）
  ・ロードマップ（短期・中期・長期）
  ・KPI設計
  ・次の仮説（学習ループへの入力）
```

## 4. 学習ループの設計

探索的システムの核心は学習ループにある。

```
問いを発見する
  ↓
仮説を立てる
  ↓
検証する（実行する）
  ↓
データを取る
  ↓
示唆を抽出する
  ↓
仮説を更新する（精度が上がる）
  ↓
また問いを発見する
```

このループが速く・正確に回るほどシステムの価値が上がる。

### ループを回すために記録するデータ

```
・どの仮説を立てたか
・何を実行したか
・結果はどうだったか
・示唆は何だったか
・次の仮説はどう変わったか

→ 全てSupabaseに保存
→ 積み上げることでシステムが賢くなる
→ SaaS化後は他ユーザーの匿名データも学習に使える
```

## 11. データモデル（Supabase）

### 主要テーブル

```
issues（問いの発見）
  id, user_id, input_text, issues_json,
  selected_issue, created_at

hypotheses（仮説）
  id, issue_id, hypothesis_text,
  priority, status, created_at

validations（検証）
  id, hypothesis_id, data_source,
  result, confidence, created_at

actions（実行）
  id, validation_id, action_text,
  status, result, created_at

learnings（学習ループ）
  id, action_id, next_hypothesis_id,
  insight_text, created_at

vols（職業図鑑）
  id, slug, occupation, parody,
  hypothesis_id, sns_score, created_at

sns_results（SNS反応）
  id, vol_id, platform, views,
  likes, comments, saves,
  completion_rate, created_at
```
